# 面试讲解

## 1. 30 秒项目介绍

这是一个本地生活服务平台后端项目。我在原有商户、优惠券、笔记、关注等业务基础上，重点改造了高并发秒杀、Redis 缓存治理、Elasticsearch 商户搜索，以及面向 Agent 的服务编排能力。

项目里秒杀链路使用 Redis Lua 做库存和一人一单的原子校验，Kafka 做异步削峰，Redisson 和数据库事务做最终兜底。商户详情使用缓存空值、互斥锁和逻辑过期处理缓存穿透与击穿。搜索能力交给 Elasticsearch，Agent 服务通过工具接口调用 Java 后端能力，生成结构化推荐结果。

## 2. 推荐讲解顺序

1. 先讲业务场景：用户浏览商户、抢优惠券、看笔记、关注用户。
2. 再讲缓存治理：为什么热点商户详情不能每次查数据库。
3. 再讲秒杀链路：高并发请求如何被 Redis 和 Kafka 拆开。
4. 再讲搜索：为什么商户搜索不适合一直用 MySQL like。
5. 最后讲 Agent：如何把后端能力包装成工具，供智能推荐服务编排。

## 3. 高频问题

### 为什么秒杀要用 Lua？

因为秒杀前置校验包含多个 Redis 操作：查库存、判断是否下单、扣库存、记录用户。如果这些命令分开发送，在并发下中间可能被其他请求插入。Lua 脚本在 Redis 服务端一次性执行，可以保证原子性。

### 为什么用了 Redis 判断一人一单，数据库还要再判断？

Redis 是前置拦截，目的是挡住大部分无效请求；数据库是最终事实来源。异步系统里消息可能重复投递，或者 Redis 数据可能被误删，所以数据库层仍然要兜底。

### Kafka 和 Redis Stream 有什么区别？

Redis Stream 适合轻量级消息队列，部署简单，和 Redis 生态结合紧密。Kafka 更适合高吞吐、日志型消息、消费者组扩展和长期消息保留。项目里引入 Kafka，是为了更贴近生产中的异步削峰方案。

### 缓存穿透、击穿、雪崩怎么区分？

缓存穿透：请求的数据数据库也不存在，导致每次都打到数据库。可以缓存空值或使用布隆过滤器。

缓存击穿：某个热点 key 失效，大量请求同时打到数据库。可以用互斥锁或逻辑过期。

缓存雪崩：大量 key 同时失效，或者 Redis 整体不可用。可以给过期时间加随机值、做多级缓存、限流降级。

### 为什么搜索要用 Elasticsearch？

MySQL 的 `like '%关键词%'` 很难利用普通索引，数据量上来后性能差。Elasticsearch 基于倒排索引，更适合关键词检索、相关性排序和复杂搜索条件。

### Agent 和普通接口有什么区别？

普通接口是前端明确调用某一个功能。Agent 更像一个编排层，它先理解用户自然语言需求，再选择调用哪些后端工具接口，最后组织成推荐结果。
