# 项目改造记录

本文档记录项目从基础本地生活业务改造成智能推荐与服务编排项目的过程，方便复盘和面试讲解。

## 阶段 0：修稳现有功能

目标：保证登录、首页、帖子详情、秒杀、Feed 等核心功能可运行。

处理内容：

- 检查登录拦截器和 token 刷新逻辑。
- 修复关注 Feed 推送时粉丝 id 取值问题。
- 修复关注 Feed 查询返回空的问题。
- 检查热门笔记未登录访问时的空指针风险。
- 检查秒杀 Lua、消息消费、订单创建事务。

收益：

项目先稳定，再继续加 Kafka、Elasticsearch、Agent，避免问题叠在一起不好排查。

## 阶段 1：缓存治理

目标：整理商户详情缓存方案，形成面试可讲的缓存治理模块。

实现内容：

- 缓存空值：解决缓存穿透。
- 互斥锁：解决热点 key 失效后的缓存击穿。
- 逻辑过期：热点数据先返回旧值，后台异步重建缓存。
- 抽取 `CacheClient`，减少重复代码。

关键点：

- Redis key 要有清晰前缀。
- 对象写入 Redis 前需要转 JSON。
- 从 Redis 读出后需要反序列化。
- 泛型方法可以让缓存逻辑复用到不同实体。

## 阶段 2：秒杀异步化

目标：把秒杀链路从同步写库改造成异步削峰。

实现内容：

- Redis Lua 前置判断库存和一人一单。
- 秒杀成功后生成订单 id。
- 发送订单消息到 Kafka。
- Kafka 消费者异步创建订单。
- Redisson 按用户维度加锁。
- 数据库再次判断一人一单并扣减库存。

关键点：

- Lua 保证 Redis 前置校验原子性。
- Kafka 负责削峰和解耦。
- 数据库仍然是最终一致性的兜底。

## 阶段 3：Elasticsearch 商户搜索

目标：新增商户搜索能力，避免把关键词搜索全部压到 MySQL。

实现内容：

- 新增 ES 配置。
- 新增商户搜索文档 DTO。
- 新增商户数据同步接口。
- 新增商户搜索接口。

关键点：

- MySQL 负责事务型主数据。
- Elasticsearch 负责搜索型查询。
- 后续可以扩展分类、评分、距离等字段。

## 阶段 4：Agent 服务编排

目标：把后端业务能力包装成 Agent 可调用的工具接口。

实现内容：

- Java 后端新增 `/agent/tools/**` 工具接口。
- FastAPI 服务调用 Java 工具接口。
- 根据用户自然语言需求返回结构化推荐结果。

关键点：

- Agent 不直接访问数据库。
- Agent 通过稳定的后端工具接口完成服务编排。
- Java 后端仍然负责业务规则、数据访问和权限控制。
