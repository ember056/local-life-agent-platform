# 架构设计

## 1. 项目定位

本项目围绕本地生活业务场景构建，核心能力包括用户登录、商户浏览、优惠券秒杀、关注 Feed、商户搜索和智能推荐服务编排。

项目的重点不是页面展示，而是后端链路设计：

- 如何用 Redis 提升高频读接口性能。
- 如何用 Lua、Kafka、Redisson 和数据库事务处理秒杀并发。
- 如何用 Elasticsearch 承担搜索能力。
- 如何把 Java 后端接口包装成 Agent 可调用的工具。

## 2. 总体架构

```text
用户 / 前端页面
    |
    v
Nginx 静态资源与反向代理
    |
    v
Spring Boot 后端服务
    |
    |-- MySQL：业务主数据
    |-- Redis：缓存、登录、点赞、Feed、秒杀预扣库存
    |-- Redisson：分布式锁
    |-- Kafka：异步订单消息
    |-- Elasticsearch：商户搜索索引
    |
    v
FastAPI Agent 服务
```

## 3. 核心模块

### 用户模块

- 手机号验证码登录。
- 登录后生成 token。
- 用户信息写入 Redis Hash。
- 请求进入后由拦截器刷新 token，并把用户信息保存到 `ThreadLocal`。

### 商户模块

- 商户详情优先查询 Redis。
- Redis 未命中时查询 MySQL。
- 查询结果写回 Redis。
- 空结果也写入 Redis，用于缓解缓存穿透。

### 秒杀模块

秒杀链路分为两个阶段：

1. 前置校验阶段：
   - Redis Lua 判断库存是否充足。
   - Redis Set 判断用户是否已经下单。
   - 校验成功后扣减 Redis 库存。
   - 生成订单消息。

2. 异步落库阶段：
   - Kafka 消费订单消息。
   - Redisson 按用户维度加锁。
   - 数据库再次判断一人一单。
   - 扣减数据库库存并创建订单。

这样做的原因是：接口可以快速返回，数据库写入压力被 Kafka 削峰，同时数据库仍然作为最终一致性的兜底。

### Feed 模块

- 用户发布笔记后，把笔记 id 推送到粉丝的 Redis ZSet 收件箱。
- ZSet 的 score 使用时间戳。
- 查询时使用滚动分页，避免普通分页在数据插入时出现重复或漏读。

### 搜索模块

- MySQL 保存商户主数据。
- Elasticsearch 保存面向搜索的商户文档。
- 搜索接口优先走 ES，适合关键词检索和后续扩展距离、评分、分类筛选。

### Agent 模块

Java 后端提供一组工具接口：

- 商户搜索
- 商户详情
- 优惠券查询
- 热门笔记

FastAPI Agent 服务负责理解用户自然语言需求，调用这些工具接口，并生成结构化推荐结果。

## 4. 数据一致性设计

- 缓存与数据库：先更新数据库，再删除缓存，降低脏数据窗口。
- 秒杀库存：Redis 负责快速预扣，MySQL 负责最终落库。
- 一人一单：Redis Set 做前置判断，数据库唯一约束或查询做最终兜底。
- 异步消息：Kafka 消费失败时可以重试，避免请求线程长时间阻塞。

## 5. 可扩展方向

- Kafka 消费失败消息进入死信队列。
- Elasticsearch 增加地理位置查询。
- Agent 服务接入真实大模型 Tool Calling。
- 增加 Prometheus、Grafana 监控。
- 增加接口级压测报告。
